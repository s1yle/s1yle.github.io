今日份疲惫加载完毕：MC插件开发的踩坑实录

此刻敲键盘的手还带着点麻木，今天是实实在在被技术问题按在地上摩擦的一天——又累又气，索性写篇随笔吐槽两句，也顺便把零散的知识点和教训记下来，免得日后再踩同款坑。

事情的起因很简单：朋友写了个MC插件，我一时兴起想凑个热闹，贡献一个小功能。本以为是半小时就能搞定的小活儿，结果硬生生耗了一整个下午，直到一小时前才总算敲完最后一个字符，发起了PR。现在回头看，今天绝大多数时间都没花在核心开发上，全耗在该死的Java环境配置上了，说多了都是泪。

前大半天项目一直跑在WSL里，各种莫名其妙的问题轮番上阵：依赖加载失败、路径映射混乱、IDEA调试断点失灵……本来就不算顺畅的开发节奏，被这些环境问题搅得一团糟。更坑的是中间还被Claude摆了一道——我本来想让它帮我理一理Git操作流程，目标很明确：同步上游分支、合并本地分支、推到远程。结果这家伙直接把main分支的内容全塞进了feature分支，瞬间引发一连串冲突，代码结构乱成一锅粥，我对着终端懵了足足十分钟，又花了好久才一点点捋顺。

最后实在没辙，咬咬牙切换到Windows环境重新配置，没想到之前卡了半天的问题全迎刃而解。只能说人有时候真不能钻牛角尖，早放弃WSL也不至于折腾这么久。顺带吐槽一句，IDEA对WSL的支持是真的拉胯，以后开发Java优先选Windows，实在不行就切双系统Ubuntu，WSL暂时先打入冷宫了。

随手记点备忘，免得转头就忘

  1. Hexo使用技巧：下载好Hexo后，先把GitHub Pages个人主页仓库clone下来，切换到dev-2分支再进行文章发布操作，避免直接在主分支乱改导致冲突。

  2. Git操作铁律：尽量保持操作简单，执行合并、推送等操作前，务必确保分支干净、配置正常，不然出了问题排查起来太费时间（血的教训）。

  3. Linux下npm权限拒绝解决方案：改变npm默认安装路径，步骤如下（亲测有效）：

    1. 创建global安装目录：mkdir ~/.npm-global

    2. 配置npm使用新目录：sudo npm config set prefix '~/.npm-global'

    3. 在/.profile文件中添加路径配置：`sudo export PATH=/.npm-global/bin:$PATH`

    4. 让配置立即生效：source ~/.profile

    5. 重新执行全局安装命令：sudo npm install -g xxxx

  4. MC服务端关系梳理：Bukkit是基础插件服务端，Spigot基于Bukkit开发且优化更好，Paper又在Spigot的基础上做了进一步优化和功能扩展，性能和稳定性更优，是目前开发插件的优选服务端。

  5. 近期开发计划：搞一个基于NapCat的QQ机器人插件，借助NcatBot SDK可以快速搭建，入门门槛不高，基础Python语法就能上手。
